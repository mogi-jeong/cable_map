# 공가 신청서 자동생성 에이전트 프롬프트

## 개요

KT 전주 공가(공동가설) 신청서를 이음 v2 케이블맵 데이터와 KT 장표(XLSX)를 기반으로 자동 생성하는 시스템.

- 소스 파일: `cable_map_gongga.js` (핵심 로직), `cable_map_ui.js` (데이터 수집/호출)
- 의존 라이브러리: SheetJS (XLSX)
- 트리거: 케이블 정보 패널 → "공가 신청서 생성" 버튼

---

## 데이터 흐름

```
케이블 클릭 → generateApplication(connId)
  ├─ 1. 전주 목록 구성 (poleList)
  ├─ 2. 전주별 장비 매핑 (equipByPoleId)
  ├─ 3. gonggaParsePoles(poleList, cableInfo) → poles[]
  ├─ 4. gonggaLoadInvs() → 장표 데이터 로드
  └─ 5. gonggaBuildApplication(poles, invs, fromNode, toNode) → XLSX 다운로드
```

---

## 1단계: 전주 목록 구성 (cable_map_ui.js)

### generateApplication(connId)

1. **conn** 객체에서 fromNode(OUT), toNode(IN) 확인
2. **전주 로딩**: waypoints의 snappedPole ID로 IndexedDB에서 전주 로드 + 장비 인근 전주 추가 로드
3. **findEquipPole(eq)**: 장비 노드로부터 30m 이내 가장 가까운 전주 탐색 (오프셋 보정 포함)
4. **poleList 구성**: startPole → waypoint 전주들 → endPole (중복 제거)
5. **장비 매핑**: 각 전주 30m 반경 내 장비 노드 탐색 → equipByPoleId 맵

### 전달 데이터 (cableInfo)

```javascript
{
  cores: conn.cores,           // 케이블 코어수 (12, 48, 72, 144 등)
  lineType: conn.lineType,     // 'new'(신설) 또는 'existing'(기설)
  equipByPoleId: {             // poleId → 장비노드[] 매핑
    "pole_id_123": [
      { type: 'junction', name: '함체명', lat, lng, ... },
      { type: 'onu', name: 'ONU명', lat, lng, ... }
    ]
  }
}
```

## 2단계: 전주 파싱 (cable_map_gongga.js)

### gonggaParsePoles(poleList, cableInfo)

각 전주 노드에서 다음 정보를 추출:

| 필드 | 소스 | 파싱 방법 |
|------|------|-----------|
| 관리구 | node.memo | `전산화번호: XXXXX` → 앞 5자리 (`/^(.{5})(\d{3})$/`) |
| 번호 | node.memo | 뒤 3자리 숫자 |
| 선로명 | node.name | `여량간-614` → `여량간` (`/^(.+)-(\d+[A-Za-z0-9]*)$/`) |
| 선로번호 | node.name | `여량간-614` → `614` (분기번호 H2, L1, L2R3 등 지원) |
| 전산화번호 | 관리구+번호 | 관리구 + 번호 3자리 패딩 → 대문자 |
| 케이블규격 | cableInfo.cores | 이음 케이블 코어수 문자열 |
| 통신선종류 | cableInfo.lineType | coax → 'C', 그 외 → 'O' |
| 장비목록 | equipByPoleId | 최대 3개, 기기코드 매핑된 것만 |

### 선로번호 regex 상세

```
/^(.+)-(\d+[A-Za-z0-9]*)$/
```

지원 형태:
- `여량간-614` → 선로명: 여량간, 선로번호: 614
- `여량간-567H2` → 선로명: 여량간, 선로번호: 567H2
- `여량간-738L2R3` → 선로명: 여량간, 선로번호: 738L2R3
- `신월간-315L21L1` → 선로명: 신월간, 선로번호: 315L21L1

### 장비 타입 → 기기코드 매핑

```javascript
var EQUIP_CODE = { 'onu': '3', 'junction': '6' };
```

| 기기코드 | 명칭 | 이음 노드 타입 |
|----------|------|----------------|
| 1 | UPS | (미구현) |
| 2 | PS | (미구현) |
| 3 | ONU | `onu` |
| 4 | 증폭기 | (미구현) |
| 5 | 탭오프 | (미구현) |
| 6 | 접속함 | `junction` |
| 7 | 분배기 | (미구현) |

### 정렬

선로번호 기준 오름차순 정렬:
1. 기본 숫자 비교 (첫 번째 숫자 그룹)
2. 분기(알파벳 포함) 전주가 본선 전에 위치
3. 같은 기본번호 내에서 문자열 비교

## 3단계: 장표 로딩

### gonggaLoadInvs(onLoaded)

KT 장표 XLSX 파일을 로드하여 두 가지 인덱스로 캐싱:

- **byId**: `시작전산화번호` → 행 배열
- **byName**: `현전주 선로명|현전주 선로번호` → 행 배열

세션 내 재사용 (\_cachedInvsData). 캐시 초기화는 페이지 새로고침.

## 4단계: 신청서 빌드

### gonggaBuildApplication(poles, invs, fromNode, toNode)

### 기본값

```javascript
var defaults = {
  설치단: '2',
  사업자: 'A000042286',   // 정선 사업자코드 (지역별 변경)
  용도: '4',
  통신선: 'O',
  규격: '12'
};
```

### 엑셀 컬럼 구조 (34열)

| 인덱스 | 그룹 | 컬럼 |
|--------|------|------|
| 0 | - | 순번 |
| 1 | - | 접수구분 |
| 2-5 | 현전주 | 선로명, 선로번호, 관리구, 번호 |
| 6-9 | 1차전주 | 선로명, 선로번호, 관리구, 번호 |
| 10-13 | 2차전주 | 선로명, 선로번호, 관리구, 번호 |
| 14-23 | 통신케이블 | 설치단, 사업자, 설치일자, 케이블번호, 용도, 통신선종류, 규격, 승인코드, 고객공급선종류, 봉인번호 |
| 24-26 | 통신기기1 | 기기코드, 사업자, 관리번호 |
| 27-29 | 통신기기2 | 기기코드, 사업자, 관리번호 |
| 30-32 | 통신기기3 | 기기코드, 사업자, 관리번호 |
| 33 | - | 비고 |

### 분류 로직

각 전주에 대해 장표 매칭 시도 (전산화번호 → 선로명|선로번호 순):

```
장표에 있음 → 정비 (해지 + 신설)
장표에 없음 → 신규
```

### 접수구분별 행 생성 규칙

#### 신설 (접수구분 2) - poleToRow2()

| 항목 | 값 |
|------|-----|
| 현전주 | pole 데이터 (선로명/선로번호/관리구/번호) |
| 1차전주 | 이전 전주 (자가주면 관리구=99999, 번호=999) |
| 2차전주 | 다음 전주 (자가주면 관리구=99999, 번호=999) |
| 통신선종류 | 이음 케이블 기준 ('O' 또는 'C') |
| 규격 | 정비: 장표 첫번째 광(O) 행 규격 / 신규: 이음 케이블 코어수 |
| 통신기기 | 전주 근접 장비 (기기코드/사업자/관리번호) 최대 3개 |

**중요: 신설 행에 동축(C) 데이터 포함 금지. 통신선종류는 항상 이음 케이블 기준.**

#### 해지 (접수구분 3) - invsToRow3()

장표 데이터 전체 그대로 복사 (광+동축 모두 유지). 전주 1개에 장표 행이 여러 개(광+동축)면 모두 해지 행으로 생성.

### 자가주 판정

```javascript
var JUMP_THRESHOLD = 2;
```

인접 전주와 선로번호 차이가 2 초과이면 자가주로 판정:
- 1차전주: 이전 전주와 비교
- 2차전주: 다음 전주와 비교
- 자가주인 경우: 관리구=99999, 번호=999, 선로명/선로번호 빈칸
- 비고란에 "1차 전주 자가주" / "2차 전주 자가주" 자동 기입

### 출력 파일

#### 신규 파일
- 시트: 시설계획서_신규 (데이터), 시설계획서_정비 (빈), 시설계획서_해지 (빈)
- 파일명: `YYMMDD_LGHV_공가신규_구간명_N본M조.xlsx`

#### 정비 파일
- 시트: 시설계획서_신규 (빈), 시설계획서_정비 (데이터), 시설계획서_해지 (빈)
- 정비 시트 구성: `▶ 해지 (접수구분 3)` 구분선 + 해지행들 + `▶ 신설 (접수구분 2)` 구분선 + 신설행들
- 파일명: `YYMMDD_LGHV_공가정비_구간명_N본M조.xlsx`

#### 본/조 계산
- 본: 고유 전주 수 (선로명-선로번호 기준 중복 제거)
- 조: 총 데이터 행 수

#### 구간명 생성
- 같은 선로: `선로명+첫번호-끝번호` (예: 여량간614-738)
- 다른 선로: `선로명A+번호A-선로명B+번호B` (예: 낙동간238-용탄지36)

## 이음 v2 데이터 모델 참조

### 노드 타입

| 타입 | 설명 | 공가 처리 |
|------|------|-----------|
| pole_existing | 기설전주 | 전주 목록에 포함 |
| pole_new | 신설전주 | 전주 목록에 포함 |
| pole_removed | 철거전주 | 전주 목록에 포함 |
| pole | 일반전주 | 전주 목록에 포함 |
| junction | 함체/접속함 | 기기코드 6 |
| onu | ONU | 기기코드 3 |
| subscriber | 가입자 | 장비매핑 제외 |
| cctv | CCTV | 장비매핑 제외 (기기코드 없음) |
| datacenter | 국사 | 장비매핑 제외 |

### 커넥션 객체

```javascript
{
  id: connId,
  nodeA: fromNodeId,     // OUT(송신)측
  nodeB: toNodeId,       // IN(수신)측
  cores: 12,             // 코어수 → 공가 규격
  lineType: 'new',       // 'new'(신설) / 'existing'(기설)
  waypoints: [           // 전주 경유 경로
    { lat, lng, snappedPole: poleId }
  ],
  portMapping: [],
  inFromCableId: null
}
```

### 전주 노드

```javascript
{
  id: poleId,
  type: 'pole_existing',
  lat: 37.xxx,
  lng: 128.xxx,
  name: '여량간-614',           // → 선로명 + 선로번호 파싱
  memo: '전산화번호: AB123456'  // → 관리구 + 번호 파싱
}
```

## 알려진 제약 / 향후 개선 사항

1. **기기코드 미구현 타입**: UPS(1), PS(2), 증폭기(4), 탭오프(5), 분배기(7) - 이음에 해당 노드 타입 추가 시 EQUIP_CODE 맵에 등록 필요
2. **사업자코드 고정**: 현재 정선 사업자(A000042286) 하드코딩. 지역 변경 시 defaults.사업자 수정 필요
3. **장표 캐시**: 세션 단위. 다른 장표 파일로 교체하려면 페이지 새로고침 필요
4. **해지 시트**: 별도 해지 파일 생성 로직 미구현 (현재 빈 시트)
5. **전주 오프셋**: 전주 미리보기 오프셋(\_polePreviewOffset)이 있을 경우 장비-전주 매핑 시 보정 적용
